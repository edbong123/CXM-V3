# DECISIONS.md
All architectural decisions recorded here as they are made. Never contradict anything in this file. If a conflict arises, flag it explicitly and stop — do not resolve it silently.

Format: decision, rationale, what was rejected and why.

---

## Dual home base: Supabase for prototype, Convex for production

The prototype phase uses Supabase for speed of setup. The production rewrite targets Convex for real-time capabilities, TypeScript-native schema, and per-org deployment isolation.

This follows the prototype/production workflow Restacked itself defines. Supabase was not rejected — it is the right tool for the prototype phase. It is not the right production architecture for a multi-tenant real-time platform.

---

## Auth: Clerk

Provides role model that maps directly to ROLES.md. Per-org configuration is first-class. Role patterns defined in Clerk are auto-generated into ROLES.md so role-aware UI is always built from the live model.

---

## Context delivery: GitMCP as the starting point

GitMCP exposes GitHub repos as live MCP endpoints. Zero infrastructure. Works with every MCP-compatible tool. This is the Level 2 entry point. Gateway (Level 4) is deferred until volume and architectural rule enforcement justify the complexity.

---

## Context lives in dedicated repos, separate from application code

Context lives in GitHub repositories separate from the application code. This allows context to be served via MCP to all AI tools independently. The application code repo is a fourth MCP endpoint, fetched on demand, not loaded at session start.

---

## ROLES.md has two sections: team roles and product roles

Team roles describe who builds the system and how. Product roles describe who uses the product and what they can do. These are distinct and maintained separately in the same file with clearly marked sections.

---

## V2 is iterative on the V1 codebase

V1 was generated by v0. V2 is being built by improving and editing the existing codebase — not starting from scratch. The Logic Builder works from the existing code with full context of what already exists. CODEBASE-MAP.md and CONVENTIONS.md are critical inputs for this work.

---

## Prototype code is never deployed to production

The Experience Builder's output is a validated specification. The Logic Builder produces the production implementation. Prototype code is never deployed. This is enforced through context, skills, and eventually gateway guardrails. It is not optional and not subject to shortcuts under deadline pressure.
