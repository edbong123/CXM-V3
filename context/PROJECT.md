# PROJECT.md
## What we are building

Restacked.ai — a context orchestration platform for AI-assisted development teams.

Current status: V2 prototype. V1 was generated by v0. V2 is being built iteratively on the same codebase — improving and editing, not restarting.

## Home base

**Prototype: Supabase**
Chosen for speed of setup during the discovery and prototype phase. PostgreSQL-native, fast to stand up, widely understood.

**Production: Convex**
The rewrite target. Chosen for real-time capabilities, TypeScript-native schema, and per-org deployment isolation — the right architecture for a multi-tenant B2B platform. Each customer gets their own deployment.

This dual home base follows the prototype/production workflow Restacked itself defines. The prototype validates logic and flow. The production rewrite rebuilds it cleanly on the right foundation.

## Team

| Role | Owns |

One person may cover multiple roles. Every role must be explicitly assigned — never left unowned.

## Hard constraints

- Prototype code is never deployed to production. It is rewritten. Non-negotiable.
- Backend architecture decisions happen in the backend project. Frontend projects never modify the data model or add business logic.
- All UI components import from the abstraction layer only. No direct database calls from UI.
- Role checks happen at the data layer, not just the UI.
- Per-org isolation tested early. One org cannot access another org's data at the infrastructure level.
- AI-generated code reviewed before production — especially auth, billing, and anything touching user access.
- Context is a PR requirement. Schema changes, role changes, and resolved decisions update context in the same PR.
- Knowledge produced in sessions translates into context files before the next development session. Context Maintainer owns this.
